### P0.2 微信小程序订阅消息机制未考虑 — 产品设计存在技术不可能

**问题**：project_plan.md 7.3优化9和technical_architecture.md 4.5均设计"每天早上8:00推送今日待办"。

**现实**：微信小程序的订阅消息（`send_subscribe_message`）要求用户**必须先主动点击订阅按钮**才能接收推送。用户不订阅则收不到任何提醒。这与"每天自动推送"的产品假设直接矛盾。

**影响**：
- 疫苗到期、复查提醒、住院日报等核心闭环功能在静默状态下完全失效
- 用户首次使用后如不主动订阅，产品退化为纯工具，失去"主动健康管理"价值

**建议**：
1. 在首次引导第4屏增加"开启提醒"步骤，引导用户一次性订阅多个消息模板
2. 在每次用户设置提醒时，若未获得订阅权限，先弹出授权申请
3. 备选方案：改用微信"公众号关联"或"服务通知"作为兜底通道

### P0.3 后端4级状态 → 前端3色映射规则缺失

**问题**：
- 后端 IndicatorEngine 返回4种状态：`normal | low | high | critical`
- UI设计仅使用3色：🟢正常 🟡注意 🔴异常
- 映射规则从未明确定义

**隐含映射**（需确认是否为设计意图）：
- `normal` → 🟢
- `low | high` → 🟡（合并为"注意"）
- `critical` → 🔴

**风险**：若 `low` 和 `high` 合并为同一UI状态，用户无法区分"偏低"和"偏高"，但在某些指标（如血压）中方向性至关重要。若分开显示，则需要4色或额外图标。

**建议**：在 functional_ui_design.md 中明确补充状态映射表，并考虑在异常详情页保留方向信息（"高了一点点" vs "低了一些"）。

## P1 — 高影响风险（不修复将显著损害用户体验或合规性）

### P1.1 色盲友好性完全缺失

**问题**：红绿灯状态语言（🟢🟡🔴）是核心交互语言，但红绿色盲（影响全球约8%男性）无法区分绿色和红色。

**影响**：用户可能将"严重异常"误判为"正常"，造成健康风险。

**建议**：在3色体系外增加辅助区分维度：
- 形状：正常=✓ 注意=! 异常=✕
- 文字标签：永远同时显示文字"正常"/"注意"/"异常"
- 增加"高对比模式"设置项

---

### P1.2 医疗影像AI处理的数据隐私盲区

**问题**：用户拍摄的检验报告原图上传至对象存储后，通过URL发送给第三方AI（Kimi/DeepSeek/OpenAI）进行OCR解析。

**缺失的考虑**：
- 第三方AI提供商是否临时存储这些医疗图像？
- 是否满足《个人信息保护法》对敏感个人信息（医疗健康数据）的处理要求？
- 用户协议中是否明确告知"报告图片将被发送至第三方AI服务"？
- 是否存在数据跨境传输（如OpenAI）？

**建议**：
1. 在隐私协议中增加"AI图像处理"独立条款
2. 优先使用国内AI提供商（Kimi/DeepSeek/通义千问），OpenAI仅作为降级选项且需用户单独同意
3. 考虑OCR预处理：在服务端对图片进行裁剪/脱敏（去除姓名、身份证号区域）后再送AI
4. 在技术文档中增加"数据隐私处理流程"章节

---

### P1.3 离线场景处理不完整

**问题**：当前设计假设用户始终在线。但医院场景（住院家属）常处于弱网环境（电梯、地下停车场、医院WiFi限速）。

**缺失设计**：
- 上传拍照后若网络断开，照片是否缓存？恢复网络后是否自动重试？
- 弱网环境下指标中心/时间轴是否显示缓存数据？缓存有效期？
- 多人同时操作同一家庭成员数据时的冲突解决策略

**建议**：
1. 在 technical_architecture.md 中增加"离线优先"章节：拍照后本地缓存→联网后自动上传→上传成功清除缓存
2. 定义数据同步策略：乐观锁（version字段）+ 最后写入优先
3. 所有列表页支持显示缓存数据+"上次更新于X分钟前"提示

---

### P1.4 删除与危险操作确认流缺失

**问题**：functional_ui_design.md 未设计任何删除确认流程：
- 删除家庭成员（及其全部历史数据）
- 删除单条报告/指标记录
- 退出家庭共享
- 删除账号（GDPR/个保法要求的数据删除权）

**风险**：误操作导致数据丢失，且账号删除流程不符合法规要求。

**建议**：补充以下页面的删除确认设计：
1. 成员删除：二次确认弹窗 + "该成员的全部X条记录将被删除" + 输入成员名确认
2. 账号注销：独立页面 + 30天冷静期 + 数据导出选项

---

### P1.5 OCR永久失败的处理流缺失

**问题**：technical_architecture.md 描述OCR任务重试3次后标记为failed，但：
- 用户在前端看到什么？
- 如何引导用户手动输入？
- 失败报告是否还保留在记录中？

**建议**：在 functional_ui_design.md P7中补充"识别失败"终态：
```
┌──────────────────────────────┐
│ 😅 这张报告没读懂             │
│ "照片有点模糊，再拍一张试试？" │
│                              │
│ [重新拍照]  [手动输入]        │
│ [保留原图，稍后再试]          │
└──────────────────────────────┘
```

### P1.6 图片压缩质量可能影响OCR准确率

**问题**：technical_architecture.md 4.5规定"客户端压缩（<2MB/Quality 0.8）"。

**风险**：医学报告中的小字（如参考范围、脚注）在0.8 JPEG压缩后可能模糊，导致AI OCR识别率下降。

**建议**：
1. 根据图片内容自适应压缩：含文字区域用更高质量（0.9+），空白区域可降低
2. 或改用服务端压缩策略：原图上传→服务端智能压缩→OCR用原图、展示用压缩图
3. 增加OCR准确率指标监控，建立质量-准确率对照表

## P2 — 中等问题（工程债务，建议修复）

### P2.1 deviation_percent 字段计算逻辑缺失

**问题**：technical_architecture.md 4.5 `ReminderService.create_from_abnormal` 使用 `indicator.get('deviation_percent', 0)`，但：
- `IndicatorEngine.judge()` 返回的是 categorical 状态（normal/low/high/critical）
- deviation_percent 的计算公式从未定义
- 该字段在数据模型（indicator_data表）中不存在

**建议**：在 `IndicatorEngine` 中增加 `calculate_deviation(value, lower, upper)` 方法，并在 indicator_data 表中增加 `deviation_percent DECIMAL(5,2)` 字段。

---

### P2.2 双写事务缺失

**问题**：上传报告后需要同时写入 `reports` 表、`indicator_data` 表、`health_events` 表。三个表的数据必须一致，但目前没有分布式事务设计。

**风险**：OCR完成后数据库写入失败会导致"有报告无指标"或"有指标无事件"的数据不一致。

**建议**：在 technical_architecture.md 中明确使用数据库事务（SQLAlchemy async session 的 `begin()`）包裹三表写入，失败时整体回滚并标记报告为 failed。

---

### P2.3 AI悬浮球拖拽与点击手势冲突

**问题**：functional_ui_design.md 描述AI悬浮球"可拖拽到屏幕任意位置（松手后缓动画回边界）"。

**风险**：用户点击悬浮球时可能被识别为拖拽，导致AI面板无法打开。微信小游戏的拖拽组件常有此问题。

**建议**：明确定义手势阈值：
- 触摸移动距离 < 10px → 识别为点击，打开AI面板
- 触摸移动距离 >= 10px → 识别为拖拽，跟随手指移动
- 松手时若移动距离 < 10px 且时间 < 200ms → 点击

---

### P2.4 长辈模式仅提及未设计

**问题**：project_plan.md Phase 4和functional_ui_design.md P3均提到"长辈模式"，但：
- 没有详细的交互设计
- 没有定义"长辈模式"具体改变哪些UI（字体大小？功能隐藏？简化语言？）
- 没有技术实现方案（是前端切换CSS变量？还是服务端存储用户偏好？）

**建议**：补充长辈模式设计文档，至少包括：
- 字体放大至1.3倍
- 隐藏儿童专区/住院专区等复杂功能
- 按钮增大至56px以上
- 语音输入优先

---

### P2.5 数据保留策略缺失

**问题**：文档未定义：
- AI对话记录保留多久？
- 已删除成员的指标数据是否物理删除？
- 报告原图是否永久保存？（当前写"永久保存，不删除原图"）
- 用户注销账号后数据如何处理？

**风险**：存储成本无限增长 + 合规风险（个保法要求明确数据保存期限）。

**建议**：
1. AI对话：保留90天，超期自动清理
2. 报告原图：保留至账号注销后1年（满足医疗记录保存要求）
3. 已删除成员数据：软删除（标记deleted_at），保留1年后物理删除
4. 在 technical_architecture.md 中增加"数据生命周期管理"章节

---

### P2.6 测试策略完全缺失

**问题**：4,229行技术文档中，没有任何关于测试策略的内容。

**缺失**：
- 单元测试覆盖率目标
- API集成测试方案
- 微信小程序自动化测试（如Minium框架）
- AI Prompt回归测试（如何确保模型更新后OCR准确率不下降？）
- 异常值边界测试（阈值边界±0.1%的case）

**建议**：在 technical_architecture.md 中增加测试章节：
```
测试策略：
- 后端：pytest + pytest-asyncio，目标覆盖率>80%
- 核心算法（异常判断/趋势评价）：100%边界覆盖
- AI：Prompt版本化 + 黄金数据集回归测试
- 前端：Minium框架E2E测试核心路径（上传→OCR→查看）
- 性能：locust压测关键API（首页加载<2s）
```

## 扩展机会（选择性采纳）

以下扩展机会按价值/成本比排序。每个扩展都附带独立的成本估算，可单独决策。

### 扩展A：增加高对比度/色盲辅助模式
- **价值**：覆盖~8%男性用户 + 提升全场景可读性（阳光直射下）
- **成本**：低（CSS变量切换 + 图标补充）
- **建议**：采纳。在"我的→设置"中增加"辅助功能"入口。

### 扩展B：离线优先架构
- **价值**：住院场景核心体验保障（医院常弱网）
- **成本**：中（需要本地SQLite/Storage封装 + 同步队列）
- **建议**：MVP后可延后，但在技术文档中预留接口设计。

### 扩展C：指标阈值版本化与多源校准
- **价值**：医学参考范围会更新（如WHO儿童生长标准更新），当前阈值硬编码
- **成本**：中（需要阈值表 + 版本字段 + 历史数据重计算策略）
- **建议**：采纳。在 indicator_engine.py 中将 THRESHOLDS 改为数据库表，支持版本管理。

### 扩展D：AI对话质量反馈闭环
- **价值**：用户可对AI回答点"👍/👎"，用于Prompt迭代和模型选择
- **成本**：低（DB表+UI按钮+后台统计）
- **建议**：采纳。在AI消息气泡下方增加 subtle 的反馈按钮。

## 审查结论

| 维度 | 评分 | 说明 |
|------|------|------|
| 产品一致性 | ⚠️ 6/10 | P0级残留矛盾影响团队执行对齐 |

**下一步建议**：

4. 将扩展A（可访问性）和扩展C（阈值版本化）纳入Scope
